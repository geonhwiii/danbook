**마이크로 프런트엔드(Micro Frontend)** 는 마이크로서비스 개념을 프런트엔드에 적용한 것으로, 하나의 거대한 앱을 **독립적으로 개발·배포 가능한 작은 프런트엔드 단위**로 쪼개 조합하는 아키텍처입니다.

## 개념

여러 팀이 각자 담당 영역(예: 검색, 장바구니, 결제)을 **독립된 앱**으로 만들고, 런타임이나 빌드 시점에 하나의 화면으로 합칩니다. 구현 방식으로는 **Module Federation**(Webpack), iframe, 웹 컴포넌트, 라우트 기반 분할 등이 있습니다.

## 장점

- **독립 배포** — 한 팀이 다른 팀을 기다리지 않고 자기 영역만 배포.
- **팀 자율성·확장성** — 팀별로 책임이 명확하고 병렬 개발이 쉬움.
- **기술 유연성** — 영역별로 다른 프레임워크/버전 사용 가능(점진적 마이그레이션 용이).
- **장애 격리** — 한 영역 문제를 전체로 번지지 않게 격리 가능.

## 단점

- **복잡성 증가** — 통합·라우팅·공통 상태 공유가 어려워짐.
- **번들 중복·성능** — 여러 앱이 같은 라이브러리(React 등)를 중복 로드하면 무거워짐.
- **일관성** — 디자인·UX를 통일하려면 별도 디자인 시스템·거버넌스가 필요.
- **운영 부담** — 빌드·배포 파이프라인과 모니터링이 복잡.

## 언제 적합한가

**대규모 조직·여러 팀**이 한 제품을 함께 만들 때 효과가 큽니다. 작은 팀·작은 앱에는 오버엔지니어링이 되어 단일 앱(모놀리식)이 더 적합할 수 있습니다.
