**RAG(Retrieval-Augmented Generation, 검색 증강 생성)** 는 LLM이 답하기 전에 **외부 지식(문서·DB)에서 관련 정보를 검색**해 프롬프트에 넣어 주는 구조입니다. 모델의 환각을 줄이고 최신·도메인 지식을 반영합니다.

## RAG 기본 흐름

```
사용자 질문 → 임베딩/검색(벡터 DB에서 관련 문서) → 검색 결과 + 질문을 프롬프트로 → LLM 생성 → 출처와 함께 응답
```

검색·임베딩·생성은 주로 백엔드/인프라 몫이지만, **사용자와 맞닿는 부분**은 프런트엔드가 책임집니다.

## 프런트엔드의 역할

- **질의 입력 UX** — 질문 입력, 필터(날짜·카테고리), 첨부 파일 업로드 등 검색 품질을 높이는 입력 설계.
- **출처·인용 표시** — 답변과 함께 **근거 문서·링크·하이라이트**를 보여 신뢰성과 검증 가능성 제공. RAG의 핵심 가치.
- **스트리밍·로딩 UX** — "문서 검색 중 → 정리 중" 같은 [단계 표시](/docs/frontend-tech-interview/66-llm-latency-ux)와 [스트리밍 출력](/docs/frontend-tech-interview/73-fetch-readablestream).
- **컨텍스트·세션 관리** — 대화 이력, [토큰 절약](/docs/frontend-tech-interview/69-token-context-management), 검색 결과 캐싱.
- **결과 표현** — 검색된 청크를 카드·인용 형태로 렌더, 관련도·신뢰도 표시, 출처로 이동.
- **피드백 루프** — 좋아요/싫어요·"이 답변이 도움이 됐나요"로 검색·생성 품질 개선 데이터 수집.
- **에러·빈 결과 처리** — 관련 문서가 없을 때 "근거를 찾지 못함"을 정직하게 표시(억지 답변 방지).
- **안전 렌더링** — 검색·생성된 콘텐츠를 [정화·검증](/docs/frontend-tech-interview/67-ai-hallucination-ui-safety).

> 한 줄 요약: 프런트엔드는 RAG의 **"입력을 잘 받고, 근거와 함께 신뢰감 있게 보여 주며, 대기·오류를 매끄럽게 처리"** 하는 사용자 경험의 책임자입니다.
