한 줄짜리 `ping` 명령에서 출발해, HTTP가 "무엇 위에 올라타 있는지"를 분해해 봅니다. `printf`와 `netcat`만으로 HTTP 요청을 손으로 보내보며 계층(layer)의 정체를 파악합니다.

## 모든 트래픽은 패킷이다

인터넷 위의 모든 통신은 **패킷(packet)** 이라는 짧은 메시지로 쪼개져 이동합니다. 각 패킷에는 보낸 사람 주소와 받는 사람 주소가 적혀 있어요. 우편엽서가 와이파이·이더넷·케이블 모뎀 위를 오가는 모습을 떠올리면 됩니다.

`ping`은 그 패킷이 출발지 → 목적지 → 다시 출발지로 왕복할 수 있는지 시험하는 도구입니다. `-c3`은 "3번만 보내고 결과 출력 후 종료".

```bash
$ ping -c3 8.8.8.8
# 3 packets transmitted, 3 received, 0% packet loss
```

`8.8.8.8`은 Google의 한 서비스 주소입니다. ping이 성공했다고 해서 **거기에 웹사이트가 있다는 뜻은 아니라는 점**이 핵심 — ping은 HTTP와 무관하게 OS 차원에서 동작합니다(ICMP echo request/reply).

## netcat + printf 로 HTTP를 손으로 보내기

웹 브라우저 없이도 HTTP 요청을 보낼 수 있습니다. `printf`는 포맷 문자열을 출력하고, `nc`(netcat)는 지정한 주소·포트로 TCP 연결을 맺어 그 문자열을 그대로 전송합니다. 둘을 파이프(`|`)로 연결합니다.

```bash
$ printf "HEAD / HTTP/1.1\r\nHost: en.wikipedia.org\r\n\r\n" | nc en.wikipedia.org 80
```

흥미로운 점: **printf는 네트워크를 모르고, netcat은 HTTP를 모릅니다.** netcat은 그냥 "포트에 연결해 바이트를 보낸다"만 할 뿐인데, 그 바이트가 마침 HTTP처럼 생겼기에 서버가 응답합니다. 이렇게 **HTTP라는 텍스트 계층** 과 **연결·전송 계층(TCP)** 이 분리되어 있음이 드러납니다.

> **실무 포인트** · HTTP/1.1에서 `Host:` 헤더는 필수입니다. 한 서버가 여러 도메인을 호스팅(Apache의 virtual host, NGINX의 server block)하기 때문에, 어느 사이트와 얘기하려는지 알려줘야 합니다.

## IETF 4계층 모델

스택의 각 계층은 바로 아래 계층에 의존하면서, 동시에 아래의 복잡함을 숨겨 위에 더 단순한 인터페이스를 제공합니다.

| 계층 | 이름                 | 대표 프로토콜           | 다루는 것                                                     |
| :--- | :------------------- | :---------------------- | :------------------------------------------------------------ |
| L4   | Application          | HTTP · SSH · SMTP · DNS | URL, 쿠키, HTTP verb, 서버 헤더 — 브라우저·웹서버가 구현      |
| L3   | Transport            | TCP · UDP               | 포트 번호, "연결(connection)"이라는 개념 — OS가 구현          |
| L2   | Internet (좁은 허리) | IP (v4 / v6)            | IP 주소와 라우팅. 위·아래 모두 IP를 거치므로 **narrow waist** |
| L1   | Hardware             | Wi-Fi · Ethernet · DSL  | 실제 비트를 같은 물리 네트워크의 장치 사이로 운반             |

인터넷 계층(L2)은 사실상 IP 하나뿐입니다. 모든 트래픽이 IP를 통과하기 때문에 "프로토콜 스택의 좁은 허리(narrow waist)"라고 부릅니다.

## 서버가 된다는 것 = 포트를 리스닝한다는 것

포트 번호는 같은 컴퓨터 위의 여러 서비스를 구분합니다. 전화기 한 대에 여러 내선번호가 있는 것과 같아요 — HTTP는 80번, SSH는 22번. `nc -l 3456`은 3456번 포트에서 연결을 기다리는(listen) 가장 단순한 형태의 서버입니다.

아무도 듣고 있지 않은 포트로 연결을 시도하면 "이 번호는 없는 번호입니다"에 해당하는 에러 패킷, 즉 **RST(reset)** 가 돌아옵니다.

- 같은 포트는 보통 한 번에 한 프로그램만 리스닝할 수 있습니다. 두 번째 시도는 에러.
- 0~1023번은 **예약 포트** — root(관리자) 권한이 있어야 리스닝 가능. 그래서 80번 웹서버를 띄울 때 `sudo`가 필요합니다.
- `sudo lsof -i`로 현재 어떤 프로그램이 리스닝 중인지(LISTEN) 확인할 수 있습니다.

## 동시 연결을 다루는 3가지 모델

실제 브라우저·서버는 수많은 연결을 동시에 처리합니다. 한 페이지를 그리려면 HTML·이미지·스크립트·CSS를 병렬로 받아야 빠르니까요. 동시성 처리 방식은 크게 셋:

- **Fork** — 연결이 올 때마다 프로세스를 둘로 쪼갬(세포 분열처럼). 오래된 웹서버, SSHD가 사용.
- **프로세스/스레드 풀** — 미리 만들어 둔 일꾼들이 하나씩 연결 담당. 빠르지만 동시 처리량에 상한이 있음.
- **이벤트 루프(단일 프로세스)** — 하나의 프로세스가 준비된 요청들 사이를 빠르게 전환. Linux의 `epoll` 등. (Node.js의 모델!)

> React/Node 개발자에게 직접 연결되는 지점입니다 — Node의 단일 스레드 이벤트 루프가 바로 세 번째 모델이에요.
