TCP가 실제로 어떻게 연결을 맺고, 데이터 순서를 보장하고, 끊는지를 `tcpdump`와 시퀀스 다이어그램으로 들여다봅니다. 3-way handshake와 TCP 플래그가 이 강의의 핵심입니다.

## 계층은 아래에 의존하고, 아래를 숨긴다

스택의 각 계층은 **바로 아래 계층의 보장(guarantee)에 의존** 하면서, 동시에 아래의 복잡함을 숨겨 위에 더 단순한 인터페이스를 제공합니다.

- 웹앱은 HTTP에 의존(URL·쿠키·verb). 하위 계층의 오작동은 보통 **에러·높은 지연** 으로 보임.
- HTTP는 브라우저·웹서버가 구현 → 이들은 OS의 **TCP** 를 사용.
- TCP는 OS 안에서 **IP** 에 의존해 패킷을 호스트 사이로 전달.
- IP는 와이파이·이더넷 같은 **하드웨어/드라이버** 에 의존.

> **용어 주의** · 같은 단어가 계층마다 다른 뜻 — Wi-Fi "connection", TCP "session(connection)", 웹 로그인 "session(쿠키)"은 서로 다른 개념입니다.

## tcpdump — 패킷의 메타데이터 엿보기

`tcpdump`는 호스트 간 네트워크 트래픽을 보여줍니다. ping을 잡아보면 ICMP echo request가 나가고 echo reply가 돌아오는 게 보여요.

```bash
$ sudo tcpdump -n host 8.8.8.8   # 특정 호스트와의 트래픽
$ sudo tcpdump -n port 53        # DNS 트래픽 (포트 53)
```

웹서버와의 통신을 잡으면 보이는 건 **패킷 헤더(메타데이터)** 뿐 — 실제 HTTP 본문은 안 보입니다. 그리고 핵심 사실: 많은 패킷의 `length`가 0입니다. 진짜 데이터를 주고받기 전후로 **연결을 세우고(setup) 끊는(teardown)** 작업이 필요하기 때문이에요.

## 시퀀스 번호와 ACK

TCP는 단순 전송을 넘어, **보낸 데이터를 상대가 받았는지 추적** 하고 순서를 보장합니다. 방법은 각 패킷에 **시퀀스 번호** 를 붙이는 것.

- 상대는 "여기까지 받았다"는 **ACK(acknowledgement)** 를 보냄
- 패킷이 사라지면 ACK가 안 와서 송신측이 알아채고 **재전송**
- 네트워크가 순서를 뒤섞어도(out-of-order) OS가 시퀀스 번호를 위치 삼아 버퍼에 **재조립**
- 시퀀스 번호는 무작위로 시작 → 다른 연결과 혼동 방지

그래서 서버가 데이터를 보낼 때, 클라이언트는 본문이 없어도(length 0) ACK 패킷을 계속 되돌려 보냅니다.

### 주요 TCP 플래그

| 플래그 | 이름        | 의미                           |
| :----- | :---------- | :----------------------------- |
| `SYN`  | Synchronize | 연결 시작, 시퀀스 번호 동기화  |
| `ACK`  | Acknowledge | "여기까지 받았다" 확인         |
| `FIN`  | Finish      | 정상적으로 연결 종료           |
| `RST`  | Reset       | 비정상 종료 / "그런 포트 없음" |

### 3-way handshake

연결을 세우는 과정은 패킷 3개로 이루어집니다:

```text
Client                          Server
  │  ── SYN ──────────────────▶  │   "연결하고 싶어"
  │  ◀──────────── SYN + ACK ──  │   "좋아, 나도 준비됐어"
  │  ── ACK ──────────────────▶  │   "확인. 시작하자"
  │                              │
  │  ◀──── 이후 데이터 교환 ────▶ │
```

## TCP 혼잡 제어 (Congestion Control)

왜 네트워크는 그냥 완벽하게 신뢰성 있게 만들지 않을까요? 빠른 네트워크 둘 사이에 느린 링크가 끼면, 한쪽이 전속력으로 쏟아부을 경우 라우터가 버퍼링하다 막혀버립니다.

그래서 TCP는 **처음엔 천천히 보내고(slow start)**, 상대의 ACK로 "따라올 수 있음"이 확인되면 속도를 올립니다. 패킷이 드롭되면 다시 늦추고, 점점 회복합니다. 라우터는 링크가 붐비면 **일부러 패킷을 드롭해 혼잡 신호** 를 보내고, 양 끝점은 속도를 줄여 응답합니다.

> **실무 교훈** · 한 서비스가 실패 시 무조건 3번 재시도하도록 짜여 있어, 과부하 때 재시도 폭주로 장애를 키운 사례가 있습니다. **네트워크는 버그 있는 앱을 구해주지 못합니다. 에러 핸들링을 신중히 설계하세요.**
