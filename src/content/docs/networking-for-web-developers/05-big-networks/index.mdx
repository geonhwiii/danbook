네트워크의 속도와 성질이 사용자 경험에 어떤 영향을 주는지 — 왜 느리고(대역폭·지연), 왜 막히는지(미들박스). `traceroute`의 영리한 트릭과 대역폭-지연 곱을 다룹니다.

## hop, traceroute, 그리고 TTL 트릭

패킷이 라우터를 한 번 거칠 때마다 그것을 **hop** 이라 합니다. `traceroute`는 목적지까지의 모든 hop(라우터 IP)을 보여줍니다.

traceroute는 원래 의도된 기능이 아니라, **무한 루프 방지용 안전장치** 를 영리하게 활용한 것입니다(Van Jacobson). 모든 패킷엔 **TTL(Time To Live)** 필드가 있어 라우터를 지날 때마다 1씩 줄고, 0이 되면 패킷이 폐기되며 **그 라우터가 출발지로 에러 메시지를 돌려보냅니다.**

- TTL=1 → 첫 라우터에서 0이 되어 폐기 → 첫 라우터 주소가 에러로 회신
- TTL=2 → 두 번째 라우터에서 폐기 → 두 번째 주소 회신
- TTL을 1씩 늘려가며 경로 전체를 재구성!

```text
TTL=1 ▶ [R1] ✕ → R1 주소 회신
TTL=2 ▶ [R1] → [R2] ✕ → R2 주소 회신
TTL=3 ▶ [R1] → [R2] → [R3] ✕ → R3 주소 회신
…
TTL=n ▶ … → [목적지] 도착 → 경로 완성
```

## 대역폭 vs 지연 — 둘 다 "느림"으로 느껴진다

사람들이 "느리다"고 할 때 보통 둘 중 하나입니다:

- **지연(latency)** — California의 Alice가 Latvia 서버에 요청하면 여러 hop·대양을 건너 왕복하느라 응답이 굼떠 보임. 거리·hop 수가 주범이고, **빛의 속도** 가 한계.
- **대역폭(bandwidth)** — 1997년 모뎀 쓰는 Bob은 100KB 로고가 15초에 걸쳐 천천히 뜸. 초당 비트 수의 문제.

여러 링크를 거치는 경로의 실효 대역폭은 **가장 느린 링크**(병목)로 결정됩니다. 서버 네트워크를 아무리 빠르게 해도 사용자의 느린 회선이 병목이면 사용자 체감 속도는 그대로예요. 단, 느린 사용자가 동시에 아주 많으면 그 합이 서버 쪽 대역폭을 넘을 수 있습니다.

> **대역폭-지연 곱 (BDP)** · `대역폭 × 왕복 지연(RTT)` = 회선에 "동시에 떠 있을 수 있는" 데이터 양. 이 값이 크면(빠르지만 먼 회선), 한 번에 충분히 많은 패킷을 쏘지 않으면 대역폭을 다 못 씁니다. TCP 윈도 크기가 이 값에 맞아야 빠른 회선이 제 속도를 냅니다.

## 미들박스 — 방화벽·프록시·NAT

**미들박스(middlebox)** 는 트래픽을 검사·수정·차단하는 별도 장치입니다(방화벽, 침입 탐지, 로드 밸런서 등).

- **방화벽** — 흔한 설정은 "허용된 (host, port)로 들어오는 것 외 전부 차단". 백엔드 DB·관리 시스템을 외부 공격에서 보호. 단, 앱 개발·배포를 방해할 수 있음. HTTP는 방화벽에서 잘 안 막혀서 많은 비-웹 앱이 HTTP를 운반 수단으로 씀.
- **필터링/검열** — 웹페이지를 에러로 바꾸기도. 학교 필터, 국가 차원 검열 등. "부적절"의 기준은 만든 사람의 편향을 반영.
- **프록시 & NAT** — 한 공인 IP 뒤에 여러 사용자가 있을 수 있음(특히 carrier-grade NAT). **IP 주소 수로 사용자 수를 세면 안 됩니다.**
