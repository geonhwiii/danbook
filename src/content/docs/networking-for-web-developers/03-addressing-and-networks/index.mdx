개별 주소를 어떻게 네트워크 단위로 묶는가(CIDR·서브넷), 주소가 사실 호스트가 아니라 인터페이스에 속한다는 것, 그리고 IPv4 주소 고갈을 NAT와 IPv6로 어떻게 버티는지를 다룹니다.

## 네트워크 prefix 와 CIDR

같은 네트워크의 주소들은 **앞부분(prefix)이 동일** 하고 특정 비트 위치부터 달라집니다. 같은 블록의 컴퓨터끼리는 라우터를 거치지 않고 직접 통신할 수 있어요.

네트워크를 말할 땐 prefix와 그 **길이** 를 함께 적습니다 — 예: `198.51.100.0/24`. /24는 "앞 24비트가 네트워크 부분"이라는 뜻이고, 남은 8비트가 호스트용입니다.

- prefix가 **길수록** 호스트 비트가 적음 → 작은 네트워크
- prefix가 **짧을수록** 호스트 비트가 많음 → 큰 네트워크
- 호스트 수 = 2^(호스트 비트) − 2 (맨 위·아래 주소 예약, 보통 첫 주소는 라우터)

서브넷 마스크는 같은 정보를 다르게 쓴 것 — 왼쪽 1, 오른쪽 0. /24 → `255.255.255.0`.

| prefix | 서브넷 마스크   | 호스트 비트 | 전체 주소 수         |
| :----- | :-------------- | :---------- | :------------------- |
| `/24`  | `255.255.255.0` | 8           | 256 (2⁸)             |
| `/16`  | `255.255.0.0`   | 16          | 65,536 (2¹⁶)         |
| `/14`  | `255.252.0.0`   | 18          | 262,144 (2¹⁸) ≈ 25만 |
| `/8`   | `255.0.0.0`     | 24          | 16,777,216 (2²⁴)     |

## 주소는 호스트가 아니라 인터페이스에 속한다

지금까지 "주소는 호스트(machine)의 것"처럼 말했지만, 더 정확히는 **주소는 인터페이스(interface)에 속합니다.** 한 호스트가 여러 인터페이스를 가질 수 있고, 각 인터페이스는 0개 이상의 주소를 가질 수 있어요.

- 유선 이더넷, 와이파이, **루프백(loopback)**, 터널, VM 가상 인터페이스 등
- 루프백은 거의 항상 `127.0.0.1` / `localhost` — 같은 호스트 안 프로그램끼리 통신
- 확인: Linux `ip addr show`, Mac/Unix `ifconfig`

**라우터** 는 서로 다른 두 IP 네트워크를 잇는 장치(게이트웨이)입니다. 보통 인터페이스가 2개 이상. 로컬 네트워크 호스트는 "바깥으로 나가는 길"인 **default gateway** 를 압니다. 집 공유기는 ISP쪽 인터페이스 1개(WAN) + 로컬 인터페이스(LAN)를 가진 아주 작은 라우터예요.

## NAT와 사설 IP — 고갈에 대한 임시방편

IPv4 주소는 약 40억 개, 사람은 70억 명 + 1인당 기기 여러 대. 그래서 ISP는 가정·사무실마다 **공인 IP 하나** 만 줍니다. 내부 기기들은 **사설 IP**(RFC 1918, 흔히 `192.168.0.0/24`)를 받고, 바깥에선 모두 그 하나의 공인 IP처럼 보입니다.

이를 가능하게 하는 게 **NAT(Network Address Translation)** — 공유기가 안↔밖 트래픽의 주소·포트를 다시 써주며 매핑 테이블을 관리합니다.

```text
사설(내부)              공인(외부)
192.168.0.11:51000  →   203.0.113.7:40001
192.168.0.12:49500  →   203.0.113.7:40002
192.168.0.11:51200  →   203.0.113.7:40003
```

> **NAT는 해결책이 아니라 우회책** · 일반 사용자가 서버를 돌리기 어렵게 만들고, 네트워크 앱 개발·디버깅을 까다롭게 합니다. "사설"은 비밀이라는 뜻이 아니라 **로컬에서만 유효하다** 는 뜻 — 전 세계 수많은 사람이 같은 192.168 주소를 각자 다른 NAT 뒤에서 씁니다.

## IPv6 — 진짜 해결책

**IPv6** 는 주소를 **128비트(16옥텟)** 로 늘려 고갈 문제를 근본적으로 해결합니다. (v5는 실험으로 끝나서 4→6.) 가장 작은 할당 단위인 `/64`조차 전체 IPv4 공간보다 큽니다.

표기는 16진수를 콜론으로 구분한 블록 — 0은 생략 가능. 예: `2607:f8b0::...`. NAT가 필요 없고 모든 기기가 고유 공인 주소를 가질 수 있습니다.

> IP 주소를 다루는 앱(로깅 포함)을 만든다면 **"주소는 32비트, dotted quad"라는 가정을 버려야** 합니다. v6가 이미 트래픽의 상당 비중을 차지합니다.
