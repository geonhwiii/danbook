사용자는 IP 주소가 아니라 이름(`www.example.net`)을 입력합니다. 그 이름이 어떻게 주소로 번역되는지(DNS), 그리고 IP 주소가 실제로는 이진수 비트열이라는 사실을 파헤칩니다.

## DNS — 인터넷의 전화번호부

**DNS(Domain Name System)** 는 이름을 주소로 바꿔주는 전 세계 분산 디렉터리입니다. 가장 잘 알려진 레코드는 **A 레코드**(Address) — `www.example.net` 같은 이름을 IPv4 주소로 매핑합니다.

DNS 클라이언트 코드(**resolver**)는 모든 OS에 내장돼 있어서, ping·nc·브라우저가 그냥 갖다 씁니다. 확인용 도구는 `host`(사람이 읽기 쉬움)와 `dig`(스크립트·설정 파일에 가까운 형식).

```bash
$ host google.com
# google.com has address 142.250.x.x

$ dig www.example.com   # question / answer / TTL 등 상세 정보
```

> **왜 중요한가** · DNS가 죽으면 사이트는 사실상 접속 불가가 됩니다. 또 DNS 도메인은 SSL 인증서·쿠키 보안의 기반이기도 합니다.

### 주요 레코드 타입

| 레코드  | 정식 명칭      | 역할                             |
| :------ | :------------- | :------------------------------- |
| `A`     | Address        | 이름 → IPv4 주소                 |
| `AAAA`  | quad-A         | 이름 → IPv6 주소                 |
| `CNAME` | Canonical Name | 별칭 → 정식 이름                 |
| `NS`    | Name Server    | 해당 도메인의 권한 있는 네임서버 |

## 캐싱과 TTL

resolver는 보통 전 세계 마스터 서버가 아니라 가까운 **캐싱 DNS 서버**(집 공유기, ISP, 혹은 Google Public DNS 등)와 대화합니다. 캐시에 없으면 root → TLD → 도메인 순으로 **재귀적으로(recursively)** 질의해 답을 얻고, 결과를 캐시에 저장합니다.

캐시는 빠르지만 문제도 만듭니다 — IP를 바꿔도 옛 주소가 캐시에 남아 있으면 일부 사용자는 옛 서버로 갑니다. 그래서 레코드마다 **TTL(Time To Live)** 이 있어 "이만큼만 캐시해라"를 지정합니다. (예: 어떤 레코드의 TTL 60초)

## IP 주소는 사실 이진수다

IPv4 주소는 **32비트(4옥텟) 값** 입니다. `241.10.27.53` 같은 점 4개 표기(dotted quad)는 그저 사람이 읽기 쉬우라고 만든 관례 — 실제 네트워크에선 32개의 비트로 전송됩니다.

각 옥텟은 8비트라서 0~255 범위입니다. 왜 255냐고요? n비트로는 2ⁿ개의 서로 다른 값을 만들 수 있기 때문입니다. 8비트 → 2⁸ = 256가지 → 0부터 255까지.

각 비트 자리의 가중치는 2의 거듭제곱입니다:

```text
비트:    1    1    0    0    0    0    0    0
가중치: 128   64   32   16    8    4    2    1
값:     128 + 64 = 192
```

중요한 직관: **주소는 수량이 아니라 "차이"를 나타냅니다.** 주소끼리 더하거나 곱하지 않아요. 우편번호처럼, 같은 네트워크 안에서 서로 달라야 한다는 게 핵심입니다.
